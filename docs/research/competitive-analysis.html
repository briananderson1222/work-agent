AnythingLLM Deep Dive

v1.11.1 MIT Node.js / Express / React

Architecture

  • 3 services: API server (Express, port 3001), React SPA frontend, Collector microservice (document ingestion)
  • SQLite via Prisma ORM (PostgreSQL optional)
  • Collector watches a hotdir/ for files, processes them, writes JSON to storage/documents/
  • Deploy: Docker, K8s, Helm, AWS CloudFormation, GCP, DigitalOcean, HuggingFace Spaces

Workspace Model

  • Workspace = isolated chat context — NOT tied to filesystem directories
  • Each workspace gets a unique slug used as the vector DB namespace
  • Per-workspace: system prompt, temperature, history window, similarity threshold, topN, chat mode (chat vs query-only)
  • Per-workspace LLM override: chatProvider + chatModel can differ from system default
  • Per-workspace agent override: agentProvider + agentModel
  • Workspace threads: sub-conversations sharing the workspace's document context
  • User scoping via workspace_users join table (admin/manager see all)

Vector DB (10 providers)

  • LanceDB (default, embedded, zero-config) — stored at storage/lancedb/
  • Chroma, ChromaCloud, Pinecone, Qdrant, Weaviate, Milvus, Zilliz, AstraDB, PGVector
  • Selected via VECTOR_DB env var
  • Abstract interface: connect(), addDocumentToNamespace(), performSimilaritySearch(), deleteVectorsInNamespace(), etc.
  • Pipeline: check cache → get embedder → split text → embed chunks → batch insert → record in SQLite
  • Optional reranking via @xenova/transformers native reranker

LLM Providers (38)

  • OpenAI, Azure OpenAI, Anthropic, Gemini, Ollama, LMStudio, LocalAI, Mistral, HuggingFace, Groq, TogetherAI, FireworksAI, Perplexity, OpenRouter, Cohere, LiteLLM, Generic OpenAI-compat, Bedrock, DeepSeek, xAI/Grok, NVIDIA NIM, SambaNova, KoboldCpp, TextGenWebUI, and 14 more
  • Each provider: streamGetChatCompletion(), getChatCompletion(), promptWindowLimit(), compressMessages()
  • 15 embedding engines (fully decoupled): native/local, OpenAI, Azure, Ollama, LMStudio, Cohere, VoyageAI, Mistral, Gemini, etc.

Document Ingestion

  • File types: PDF, DOCX, PPTX, XLSX, ODT, ODP, EPUB, CSV, JSON, HTML, MD, RST, TXT, MBOX
  • Media: MP3, WAV, MP4 (Whisper transcription), PNG/JPG/WEBP (OCR)
  • Data sources: YouTube transcripts, recursive web scraping, Confluence, Obsidian vaults, Paperless-ngx, Drupal, GitHub/GitLab repos
  • Background sync jobs for watched documents

Agent System

  • Custom "AIbitat" framework (AutoGen-inspired, multi-agent)
  • Skills: memory, doc summarizer, web scraping, web browsing, SQL agent (MySQL/PG/MSSQL), charts
  • MCP client: any MCP server's tools auto-exposed as agent functions
  • Agent Flows: JSON-defined multi-step workflows
  • No coding/IDE features
Steal from AnythingLLM: Workspace-as-vector-namespace isolation pattern, LanceDB as embedded default, per-workspace LLM override, decoupled embedding engine selection, the VectorDatabase abstract interface.

t3code Deep Dive

v0.0.4-alpha MIT Bun / Effect-ts / React / Electron

Architecture

  • Turborepo monorepo: apps/server, apps/web, apps/desktop, apps/marketing
  • Server: Node.js WebSocket + HTTP, published as npx t3 (port 3773)
  • Web: React 19 + Vite + TanStack Router/Query
  • Desktop: Electron 40 shell wrapping the web app
  • Effect-ts used pervasively for DI, error handling, service composition
  • SQLite (@effect/sql-sqlite-bun) for persistence
  • Data flow: Browser ↔ WebSocket ↔ Node server ↔ codex app-server (JSON-RPC over stdio)

Project Model

  • Project = filesystem directory (workspaceRoot path)
  • Schema: { id, title, workspaceRoot, defaultModel, scripts[], createdAt }
  • Threads scoped to projects, each thread can have a branch and worktreePath
  • File search: fuzzy search within project root (respects .gitignore, max 25k entries, 15s cache)
  • Auto-bootstrap: creates project from cwd on startup
  • Configurable scripts per project (run/test/lint/build/debug)

Provider System

  • Currently Codex-only — ProviderKind = Schema.Literal("codex")
  • claudeCode reserved in contracts but not implemented
  • Models: GPT-5.4, GPT-5.3 Codex, GPT-5.3 Codex Spark, GPT-5.2 Codex, GPT-5.2
  • Adapter pattern: ProviderAdapterShape interface with startSession, sendTurn, interruptTurn, respondToRequest, subscribeToEvents
  • Codex integration: spawns codex app-server as child process, JSON-RPC over stdio
  • Runtime modes: full-access (no approval) vs approval-required (prompts for file/command approval)

Coding UX

  • No embedded code editor — the AI agent edits code, not the user
  • Diff viewer: @pierre/diffs library, stacked/split view, per-turn and full-thread diffs, virtualized rendering
  • Terminal: Real PTY via node-pty (or Bun native PTY), xterm.js in browser, thread-scoped, history persistence
  • File operations: fuzzy file search, writeFile, open in external editor (VS Code, Cursor)
  • Composer: Lexical-based rich text, @mentions for file references, slash commands, image attachments
  • Git: branch/checkout/create, worktree create/remove, pull, stacked actions (commit → push → PR), AI-generated commit messages
  • Checkpointing: git-based checkpoints after each agent turn, diff blobs in SQLite, revert support

Event Sourcing (CQRS)

  • Full event sourcing: orchestration_events table with sequence, aggregate_kind, stream_id, event_type, payload
  • Decider: pure function decideOrchestrationCommand(command, readModel) → events[]
  • Projector: folds events into read model, writes to projection tables
  • Reactors: side-effect handlers (start provider session, create checkpoint, etc.)
  • Client state: Zustand store hydrated from server snapshot, updated via WebSocket push

Desktop vs Web

  • Web: React SPA served by server, all calls via WebSocket RPC
  • Desktop: Electron spawns t3 server as child process, custom t3:// protocol
  • NativeApi abstraction: checks for window.nativeApi (desktop injection) first, falls back to WebSocket
  • Desktop adds: native folder picker, confirm dialogs, context menus, auto-updater
Steal from t3code: Directory-based project model, diff viewer per agent turn (with @pierre/diffs), real PTY terminal (node-pty + xterm.js), git worktree support for parallel sessions, checkpoint/revert pattern, @mention file references in composer, approval workflow for agent actions.
Note on terminal-first approach: t3code spawns codex app-server directly as a child process and communicates via JSON-RPC over stdio. This is more brittle than a protocol abstraction (like ACP) but significantly more powerful — it gives direct control over the agent process lifecycle, real-time event streaming, and the ability to integrate any CLI tool that speaks stdio. This is arguably the more pragmatic path for coding agent integration.

Open WebUI Deep Dive

v0.8.8 BSD-3 Python / FastAPI / SvelteKit

Architecture

  • Backend: Python 3.11+, FastAPI, SQLAlchemy + Alembic, Uvicorn
  • Frontend: SvelteKit 2 + Svelte 5, Tailwind v4, Vite
  • DB: SQLite (default), PostgreSQL optional
  • Auth: JWT, OAuth, LDAP, SCIM 2.0
  • Real-time: Socket.IO + Redis
  • Storage: local filesystem, S3, GCS, Azure Blob
  • Observability: OpenTelemetry
  • Massive codebase: config.py is 130KB, middleware.py is 197KB, retrieval.py is 123KB

"Workspace" Concept

  • No real workspace/project isolation
  • "Workspace" = UI permission flags: WORKSPACE_MODELS_ACCESS, WORKSPACE_KNOWLEDGE_ACCESS, WORKSPACE_PROMPTS_ACCESS, WORKSPACE_TOOLS_ACCESS
  • Controls whether non-admin users can access the "Workspace" section of the UI
  • Chat organization via folders (purely UI groupings, no backend isolation)
  • Access control via generic access_grants ACL table

Vector DB / RAG (11 providers)

  • Chroma (default, embedded), Milvus, Qdrant, Pinecone, Elasticsearch, OpenSearch, PGVector, Oracle23ai, S3Vector, Weaviate, OpenGauss
  • Embeddings: sentence-transformers (default/local), OpenAI, Ollama, Azure OpenAI
  • Hybrid search: vector + BM25 via EnsembleRetriever
  • Reranking: CrossEncoder, ColBERT, external API
  • 20+ web search providers for web RAG
  • Per-file collection: file-{file_id}, per-knowledge-base: knowledge-{id}
  • Milvus/Qdrant have multitenancy modes (deployment-level, not per-user)

Multi-Provider LLM

  • Ollama: multi-instance support (OLLAMA_BASE_URLS semicolon-separated), load balancing, full API proxy
  • OpenAI-compatible: multi-endpoint (OPENAI_API_BASE_URLS + OPENAI_API_KEYS), works with LMStudio, Groq, Mistral, OpenRouter, etc.
  • Azure OpenAI: native support via DefaultAzureCredential
  • Anthropic: detected by URL pattern, uses anthropic SDK
  • Google Gemini: google-genai SDK
  • Direct connections: users can add their own API connections in UI
  • Pipelines: external proxy servers for custom logic, function calling, rate limiting

Knowledge Management

  • Named knowledge bases with file management
  • Schema: { id, user_id, name, description, meta } + KnowledgeFile junction table
  • CRUD + batch file add + export as ZIP
  • Access control via access_grants (share with users/groups, read or read+write)
  • Each knowledge base gets its own vector collection: knowledge-{id}
  • Knowledge base metadata also embedded into shared knowledge-bases collection for semantic search across bases

Code Execution

  • Jupyter: JupyterCodeExecuter connects to external Jupyter server, executes Python, captures stdout/stderr/images
  • Pyodide: Python in browser via WebAssembly (client-side, no server round-trip)
  • RestrictedPython: sandboxed in-process Python for user-defined tools
  • Terminal proxy: proxies HTTP + WebSocket to admin-configured terminal servers
  • CodeMirror: editor for writing custom tools/functions
  • MCP client: connects to external MCP tool servers
Steal from Open WebUI: Knowledge base model with ACL + file management, hybrid search (vector + BM25 + reranking), multi-instance Ollama with load balancing, multi-endpoint OpenAI-compatible pattern, semantic search across knowledge bases, user-level direct connections.
Warning: Open WebUI is a monolith with some truly massive files (130KB config, 197KB middleware, 123KB retrieval router). It's feature-rich but architecturally brittle. Don't try to port patterns wholesale — extract the concepts.

Goal 1: Workspace → "Project" Concept

Your current WorkspaceConfig is a UI layout — tabs, components, prompts. That's valuable, but it's not a "project." Here's the proposed model:

Proposed: Two-Level Hierarchy

ConceptWhat it isInspired by
Project (new)A directory-bound context with its own vector DB namespace, LLM config, document collection, and agent connections. The isolation boundary.t3code's workspaceRoot + AnythingLLM's workspace isolation
Layout (renamed from Workspace)A UI arrangement of tabs/components/prompts. Multiple layouts can exist within a project. Think "views" into the same project context.Current stallion WorkspaceConfig

Key properties of a Project:

  • workspaceRoot — filesystem directory (optional, for coding projects)
  • vectorNamespace — slug used as vector DB namespace for document isolation
  • provider / model — per-project LLM override (falls back to system default)
  • embeddingProvider / embeddingModel — per-project embedding config
  • layouts[] — array of layout configs (your current workspace tabs)
  • documents[] — uploaded/ingested documents scoped to this project
  • agents[] — agent configs or ACP connections scoped to this project

Cross-project queries: This is stallion's unique advantage. The sidebar shows all projects. A "global" chat can fan out queries to multiple project vector namespaces. None of the three competitors do this well.

ACP integration: A project with a workspaceRoot could auto-configure an ACP connection that passes the directory as cwd. Or — taking the t3code approach — spawn a coding agent directly as a child process with that directory as its working dir. The latter is more direct and powerful.

Goal 2: Vector DB Integration

Recommended Approach

  • Default: LanceDB (embedded) — zero config, stores in .stallion-ai/vectordb/. This is what AnythingLLM uses and it just works. No external service needed.
  • Provider interface: IVectorDbProvider with connect(), addDocuments(namespace, chunks), search(namespace, query, topK), deleteNamespace(namespace). Fits your existing provider registry.
  • Per-project namespace: project slug = vector DB namespace. Documents uploaded to a project are embedded and stored in that namespace only.
  • Embedding engine as separate provider: IEmbeddingProvider decoupled from LLM. Default to local (@xenova/transformers), with options for Ollama, OpenAI, Bedrock Titan embeddings.
  • Implement as a plugin: @stallion-ai/plugin-vectordb registers both providers. Core just defines the interfaces.

Implementation Pattern

  • You already depend on the Vercel AI SDK (ai package). It has provider adapters for OpenAI, Anthropic, Google, Mistral, and more. Lean on this rather than writing raw HTTP clients.
  • Each provider = a plugin that registers an ILLMProvider. Core defines the interface.
  • Per-project override: project config specifies { provider: "ollama", model: "llama3.2" }. Falls back to system default (Bedrock).
  • Open WebUI's multi-instance Ollama pattern is worth stealing: OLLAMA_BASE_URLS as a semicolon-separated list with load balancing.

Goal 4: Coding Capabilities

The t3code Insight

t3code's key insight: the UI doesn't edit code — the agent does. The UI shows you what the agent changed (diffs), lets you approve/reject, and gives you a terminal to verify. The user's job is to direct and supervise, not to type code.

This maps well to stallion's existing patterns:

  • Tool approval flow → agent action approval (file writes, command execution)
  • SSE streaming → real-time agent output
  • ACP → external coding agent connection (or direct child process spawn à la t3code)

On ACP vs direct child process: t3code spawns codex app-server directly as a child process and communicates via JSON-RPC over stdio. This is more brittle than ACP's protocol abstraction but significantly more powerful — direct control over process lifecycle, real-time event streaming, and the ability to integrate any CLI tool that speaks stdio. For coding specifically, the direct approach is probably the right call. ACP remains valuable for connecting to remote or third-party agents, but for local coding agents, a direct spawn is simpler and faster.

Cross-Project Advantage

This is where stallion beats all three competitors. None of them let you easily work across projects:

  • AnythingLLM: workspaces are isolated silos. No cross-workspace queries.
  • t3code: one project at a time. Switch projects = switch context entirely.
  • Open WebUI: no project concept at all.

Stallion's sidebar shows all projects. You can have a coding session in Project A, switch to Project B's knowledge base to ask a question about their API docs, then switch back. Or fire off a query that searches across all project vector namespaces. This is the "unified cockpit" that none of the others provide.

Fork or Build?

Verdict: Don't Fork. Build.

None of the three are good fork candidates:

ProjectWhy not fork
AnythingLLMExpress monolith, no plugin system, no SDK pattern, no desktop/mobile story beyond Docker. You'd rip out the architecture to add what stallion already has.
t3codev0.0.4 alpha, Codex-only, Effect-ts everywhere (massive learning curve, different paradigm), no vector DB, no multi-provider, no plugin system. You'd build 80% from scratch anyway.
Open WebUIPython/SvelteKit (completely different stack), monolithic (130KB config file), no plugin system, no real workspace isolation. Tech stack mismatch kills it. Also: custom license with branding restriction clause — cannot remove "Open WebUI" branding in deployments >50 users without enterprise license.

Stallion's architecture is better positioned because:

  • Plugin system — providers, vector DBs, coding features can be added modularly without touching core
  • SDK pattern — @stallion-ai/sdk means community can build on top
  • Provider registry — singleton/additive pattern already designed for this extensibility
  • Multi-platform — web, Tauri desktop, Android already working

Recommended Build Order

Phase 1: Multi-Provider Plugin

Highest ROI ~1-2 weeks

  • Ollama provider (LLM + embeddings) — unlocks fully-local story
  • OpenAI-compatible provider — covers dozens of providers with one adapter
  • Leverage Vercel AI SDK provider adapters you already depend on
  • System-level default + per-agent override (extend to per-project later)

Phase 2: Project Concept

Data model change ~1-2 weeks

  • Rename WorkspaceConfig → LayoutConfig (or keep "workspace" for the UI arrangement, add "project" above it)
  • Introduce ProjectConfig: workspaceRoot, provider/model override, vectorNamespace
  • Projects contain layouts. Sidebar shows projects with expandable layouts.
  • Migration path: existing workspaces become layouts within a "Default" project

Phase 3: Vector DB Plugin

Knowledge layer ~2-3 weeks

  • LanceDB as embedded default (zero config)
  • IVectorDbProvider + IEmbeddingProvider interfaces in core
  • Document upload UI in project settings
  • Per-project namespace isolation
  • RAG integration: inject relevant chunks into agent context

Phase 4: Coding Layout

Differentiation ~3-4 weeks

  • Built-in layout with: chat + diff viewer + terminal + file tree
  • Real PTY via node-pty + xterm.js
  • Diff viewer via @pierre/diffs or similar
  • Direct child process spawn for coding agents (t3code pattern) alongside ACP
  • Approval workflow for agent file writes and command execution

Second-Pass: UX Patterns Worth Stealing

From AnythingLLM

Sidebar & Navigation

  • Drag-to-reorder workspaces via react-beautiful-dnd, order persisted to localStorage (zero server calls)
  • Hover reveals upload + settings icons per workspace
  • Debounced search (500ms) across workspaces AND threads
  • "Virtual active" workspace on home page via LAST_VISITED_WORKSPACE localStorage
  • Mobile: full-screen overlay sidebar with dark backdrop

Document Upload UX

  • Split-panel document manager: left = all available docs, right = docs in this workspace, separated by arrow icon
  • Drag-and-drop zone + URL scrape input
  • Embedding cost preview before confirming (OpenAI only)
  • Data connectors tab: GitHub, GitLab, YouTube, Confluence, Obsidian, web scraping
  • Checks if collector service is online, shows warning if down

Workspace Settings

  • 5 tabs: General, Chat, Vector Database, Members, Agent
  • Inline LLM switcher: searchable split-panel modal (providers left, models right) accessible from chat toolbar — change provider mid-conversation
  • Per-workspace: system prompt, temperature, history window, similarity threshold (4 presets), chat mode (chat vs query-only)
  • Floating "Update Workspace" CTA only appears when form is dirty (hasChanges pattern)

Agent & Commands

  • @ in textarea triggers agent menu (also via toolbar button)
  • / triggers slash commands: reset chat, end agent session, user-defined presets
  • Agent shows ability tags: rag-search, web-scraping, web-browsing, save-file, chart-generation
  • Embed widget: admin creates per-workspace embeds with domain allowlists, rate limits, and toggles for model/temp/prompt override

From t3code

Layout & Navigation

  • Two-panel layout: left sidebar (project-grouped thread list) + right content area
  • Projects are collapsible sections in sidebar — no separate project switcher
  • Threads show status pills: "Working" (pulsing), "Completed", "Pending Approval" (amber)
  • PR status icons on threads (open/closed/merged)
  • Terminal running indicator (teal pulsing icon)
  • Right-click context menu: Rename, Mark unread, Copy ID, Delete

Diff Panel

  • Turn chip strip: horizontal scrollable chips — "All turns" + one per completed turn. URL-driven (?diff=1&diffTurnId=...) so diffs are bookmarkable
  • Stacked (unified) or split view toggle
  • Virtualized rendering via @pierre/diffs
  • Click file header → opens in external editor
  • Three render modes: inline sidebar, sheet (mobile), resizable right sidebar

Terminal

  • Bottom drawer with drag-to-resize (180px to 75% viewport)
  • Multi-terminal: split view (side-by-side grid) + tab groups with tree-style sidebar
  • Per-thread terminal state (persisted via Zustand)
  • Clickable file paths and URLs in output
  • Theme sync via MutationObserver on document class changes
  • 5000 line scrollback, SF Mono font

Approval Flow

  • Hijacks the composer: text input replaced by approval panel in-place (not a modal)
  • Shows: "PENDING APPROVAL" + summary + counter (1/N)
  • Four actions: Cancel turn, Decline, Always allow this session, Approve once
  • Composer input disabled during approval state
  • Request kinds: command, file-read, file-change

Composer / @mentions

  • Lexical-based rich text editor with custom ComposerMentionNode
  • @ triggers fuzzy file search (120ms debounce) → inserts atomic chip with VSCode file icon
  • Chips are non-editable, arrow keys skip over them, backspace deletes whole chip
  • /model → switch model, /plan → plan mode, /default → normal mode
  • Image attachments: drag-and-drop/paste, thumbnail strip below editor

From Open WebUI

Multi-Model Parallel Responses

  • Model selector supports MULTIPLE simultaneous models — selectedModels is an array
  • + button adds model slots (permission-gated)
  • Get side-by-side responses from different models in the same conversation
  • Models can be pinned to user settings
  • Fuse.js fuzzy search over model list

Knowledge Base Attachment

  • Two paths: # command in message input (autocomplete) OR paperclip menu → Knowledge tab
  • Paginated list of knowledge bases, expandable to individual files
  • Can attach whole collection or single file
  • Injected as file attachment into the message context

Channels (Slack-like)

  • Real-time messaging via WebSocket (Socket.IO)
  • Threaded replies in resizable side pane (mobile = drawer)
  • Typing indicators (5s timeout), reactions, unread tracking
  • Optimistic UI with temp_id UUIDs
  • DM support, webhooks per channel
  • Unique feature — none of the others have this

Arena / Model Evaluation

  • Admin configures "arena models" — virtual models that randomly route to real models for blind comparison
  • Users rate responses (thumbs up/down)
  • Ratings feed an ELO leaderboard — rank, rating, win/loss, sortable
  • Per-model usage analytics over time
  • Great for deciding which provider/model to default to

Tools / Code Editor

  • Monaco-based code editor for writing custom Python tools
  • Pre-filled boilerplate with Tools class and example methods
  • "Valves" = runtime config per tool via dynamic JSON schema form
  • Import tools via URL or JSON file
  • Tools are stored in DB, executed server-side via RestrictedPython sandbox

Admin Panel

  • 13 settings tabs: general, connections, models, evaluations, integrations, documents (RAG), web search, code-execution, interface, audio (STT/TTS), images (DALL-E/ComfyUI), pipelines, database
  • Additional routes: users, evaluations (leaderboard), functions, analytics
  • Comprehensive but overwhelming — stallion's plugin approach is cleaner

1. Vision & What It Delivers

Project Stallion is a complete AI agent platform — not just a chatbot, but a full system for defining, running, and interacting with multiple AI agents powered by Amazon Bedrock. It ships as five delivery surfaces from a single codebase:

💻

VoltOps Console

Core Capabilities

🤖 Multi-Agent Runtime

Define agents as JSON files with system prompts, models, tools, and guardrails. Load and switch between agents at runtime — no server restart needed. Each agent gets isolated memory, tools, and configuration.

🔧 MCP Tool Orchestration

Automatic MCP server lifecycle management. Agents declare which MCP servers they need; the runtime handles spawning, health checks, tool name normalization (for Bedrock Nova compatibility), and approval workflows.

🧩 Plugin Architecture

Workspace plugins are npm packages that import only from @stallion-ai/sdk. They provide custom UI components (Calendar, CRM, dashboards) that render in the main canvas while chat lives in a collapsible dock.

💾 Local-First Storage

File-based NDJSON storage via a custom VoltAgent StorageAdapter. Conversations, messages, working memory, and workflow states all persist to disk. Designed for clean adapter swaps to cloud backends later.

🌊 Streaming Pipeline

Handler-chain architecture for real-time AI responses. Reasoning blocks, tool calls, tool approvals, text deltas, and metadata all flow through composable pipeline handlers on both server and client.

📐 Configuration-Driven

Everything is defined in files under .work-agent/: agents, tools, workspaces, app config. JSON schemas validate all configuration. No database required to get started.

WorkAgentRuntime — Key Responsibilities (2,859 lines)

Agent Lifecycle

  • initialize() — Loads app config, creates BedrockModelCatalog, UsageAggregator, creates default agent (always available, no tools), loads all agents from disk, sets up VoltAgent with Hono server
  • reloadAgents() — Hot-reload: detects added/removed agents, cleans up MCP connections for deleted agents, creates new VoltAgent instances for added agents
  • createVoltAgentInstance() — 10-step pipeline: load spec → create Bedrock model → create memory adapter → load MCP tools → cache tools → add to global registry → calculate fixed token counts → create approval hooks → process template variables → create Agent
  • switchAgent() — Runtime agent switching without server restart
  • Default agent always created with defaultModel and no tools

Bedrock Integration

  • BedrockModelCatalog — Lists foundation models, inference profiles, fetches pricing from AWS Pricing API (paginated). Caches all three. Resolves inference profile IDs (auto-prefixes us. for profile-only models).
  • calculateCost() — Uses AWS Pricing API with fallback defaults ($0.003/1K input, $0.015/1K output)
  • calculateContextWindowPercentage() — 200K default context window
  • Model override caching — creates cached agent variants keyed as slug:modelId

Infrastructure

  • CORS — Allows localhost:* origins, configurable via ALLOWED_ORIGINS env var
  • Health Checks — 60-second interval, checks agent loaded/model/memory/MCP status
  • Monitoring Events — agent-start and agent-complete events with timestamp, traceId, usage, artifacts, steps, toolCallCount
  • Template Variables — replaceTemplateVariables() processes system prompts with 10 built-in variables ({{date}}, {{time}}, {{datetime}}, {{iso_date}}, {{iso_datetime}}, {{timestamp}}, {{year}}, {{month}}, {{day}}, {{weekday}}) plus custom variables from AppConfig
  • Swagger UI — Available at /ui via VoltAgent's honoServer

Conversation Stats Engine (onEnd hook)

The createToolApprovalHooks method contains the entire per-conversation stats tracking system in its onEnd hook:

  • Incremental calculation — Only computes new user message tokens each turn, not all messages. Uses cached fixed token counts (system prompt + MCP tools) calculated once at agent init.
  • Token breakdown — systemPromptTokens, mcpServerTokens, userMessageTokens, assistantMessageTokens → stored in conversation metadata
  • Per-model stats — Tracks input/output tokens, turns, tool calls, and cost per model within each conversation (supports mid-conversation model switching)
  • Message enrichment — After each turn, enriches the last assistant message with model capabilities (modalities, streaming support) and pricing data from AWS Pricing API
  • Cost tracking — Accumulates estimated cost per turn using live pricing

Tool Elicitation Wrapper

wrapToolWithElicitation() wraps non-auto-approved tools. On denial, returns {success: false, error: 'USER_DENIED'} to the LLM (doesn't throw) so the model can ask the user what to do differently. Auto-approved tools skip the wrapper entirely. Supports wildcard patterns (*, prefix*).

9. Streaming Pipeline

The streaming architecture is a dual-layer pipeline — server-side handlers process the raw Bedrock stream, and client-side handlers render it in the UI.

Server-Side Pipeline

StreamPipeline chains handlers via .use(handler). Each handler implements async *process(input): AsyncGenerator<StreamChunk>.

Bedrock Stream→ ReasoningHandler→ TextDeltaHandler→ ToolCallHandler→ ElicitationHandler→ MetadataHandler→ CompletionHandler→ SSE
HandlerResponsibility
ReasoningHandlerDetects <thinking> tags in text stream, splits into reasoning vs. text content parts. Character-by-character parsing with tag buffering.
TextDeltaHandlerPasses through text delta chunks
ToolCallHandlerTracks tool call lifecycle (start → args → result)

  • VoltAgentAdapter — Converts VoltAgent chunk format to the internal BedrockChunk type

Chat Endpoint SSE Wire-Up

The POST /api/agents/:slug/chat endpoint (~500 lines) orchestrates the full streaming flow:

SSE Headers→ InjectableStream→ Elicitation Callback→ AbortController→ Conversation Create→ agent.streamText()→ Pipeline.run()→ SSE Write→ [DONE]
  • Elicitation — Checks auto-approve patterns (both normalized and original tool names). Non-approved tools inject tool-approval-request events into the stream and block on a Promise with 60-second timeout.
  • Model Override — Validates model ID via BedrockModelCatalog, creates cached agent variant keyed as slug:modelId
  • Abort Handling — AbortController tied to client disconnect signal. Saves cancellation message if no output was produced.
  • Monitoring — Emits agent-start and agent-complete events with traceId, usage, artifacts, steps, toolCallCount
  • SSE Format — data: {JSON}\n\n with setTimeout(0) flush after each chunk
